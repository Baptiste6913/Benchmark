## 4. Conflits doctrinaux à régler (non triviaux)

### 4.1 Qui orchestre ? (benchmark-lead vs mainthread)

| | main v1.2.2 | v1.1.x hardening |
|---|---|---|
| Orchestrateur | **mainthread** Claude Code (envoie N Agent calls en 1 message) | `benchmark-lead` agent |
| benchmark-lead rôle | cadrage (début) + consolidation (fin) | orchestre les 9 phases |
| Parallélisation | par le mainthread | par benchmark-lead |
| Référence | `docs/agents-dispatch-investigation.md` + `workflows/orchestrate-benchmark.md` (v1.1.0) | `docs/architecture.md` (v1.1-hardening §2.2) |

Le Sprint 1 de main a formalisé que **benchmark-lead ne peut PAS dispatcher d'autres sub-agents** (limite native Claude Code). Cette contrainte technique n'a pas bougé en 2026 — elle reste vraie. Donc la doctrine v1.1 (benchmark-lead orchestre) est **cassée en pratique**, et main a raison.

**Décision** : garder la doctrine main. Tous les patchs v1.1 qui disent "benchmark-lead fait X" doivent être **retraduits** en :
