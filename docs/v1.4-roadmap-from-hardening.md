### 1.2 Lot 3 — Nouvelles briques méthodo (effort total ~2-3h)

- **`patch(2) 38fa6a6` — skill `entity-normalizer`**
  Nouveau skill entre `market-scanner` et `vendor-deep-dive` pour dédoublonner
  les solutions nommées différemment. Apporte :
  - Clustering Levenshtein normalisée + match de domaine
  - 1 WebSearch de confirmation par cluster (économie volontaire)
  - Fusion si `entity_confidence: high|medium`, sinon `needs_human_review: true`
  - Output `discovery/canonical-solutions.json`

  **Effort : M (1h30-2h)**. **Dépendance** : aucune code-wise, mais apporte de
  la valeur opérationnelle visible dès le second run.

- **`patch(5) 721158c` — `freshness_days` par critère + verdict STALE**
  Modifie les 3 grilles legacy. Le champ `criterion.freshness_days` est déjà
  accepté par le schéma depuis v1.2.2. Le fact-checker gagne un verdict `STALE`
  qui re-vérifie automatiquement si `age_days > freshness_days`. Les champs de
  `vendor-deep-dive` passent au format structuré
  `{value, source_url, verified_on, reliability}`.

  **Effort : M**. **Dépendance** : aucune. **Combo recommandé** avec la
  migration `label_fr` des 3 grilles legacy (déjà BACKLOG) puisque les deux
  modifient les mêmes fichiers.

- **`fix(2) 8e83888` — règle critic N/D systémique + `sourcing_hints`**
  Modifie `agents/critic.md` pour ajouter la règle "si un critère est N/D sur
  >50% des vendors, c'est un N/D systémique = défaut de sourcing, pas de
  vendor". Ajoute le concept `sourcing_hints` aux critères de grille (déjà
  accepté par le schéma v1.2.2 et présent dans `public-sector-fr.json`).

  **Effort : S**. **Dépendance** : aucune.

### 1.3 Lot 4 — Features à redimensionner après un vrai run (effort total ~3-5h)

- **`patch(7) 2470075` — agent `scope-clarifier` dédié**
  **Décidé v1.3 Q3 : fold dans `benchmark-lead`.** L'option de créer un agent
  séparé reste ouverte si un second bench réel révèle une friction avec le
  modèle fold (benchmark-lead alourdi en cadrage). **Effort : M**. **Décision
  à re-valider après 1 ou 2 runs v1.3**.

- **`patch(9) 7e3285f` — memory bank cross-mission**
  Arborescence `memory-bank/` (gitignored) + `memory-bank-public/` (committable)
  pour les fiches éditeurs réutilisables entre missions. Workflow investigator
  consulte avant `vendor-deep-dive`, merge à la fin avec bloc `history`.
  **Effort : L (3-4h)**. **Conflit conceptuel avec `bench.json` canonique** :
  les factsheets vivent dans `bench.actors[]` — où se place la memory bank ?
  **Décision à trancher** : (a) fiches memory-bank sont des "templates" qu'on
  copie dans bench.actors[] lors d'un nouveau run, (b) memory bank est un
  cache séparé consulté en prélude. **Effort : à redimensionner après une
  vraie mission avec 2 runs consécutifs sur des marchés qui se chevauchent**.

- **Docs architecture.md complète (`5900e9b`, 363 lignes)**
  Le document v1.1-hardening est un deep dive architecture. Une partie est déjà
  couverte par `docs/run-persistence.md` (ajouté en v1.3 patch 3) et par le
  README mis à jour Sprint 1. Le reste (principes, layers, extension points,
  glossaire) serait utile mais nécessite **adaptation au modèle bench.json
  canonique** (le doc original parle de `factsheets/*.json` + `scoring/scores.json`
  distribués, ce qui n'est plus la source de vérité). **Effort : M (1-2h)**.

- **`--resume <run_id>` flag actif** (partie exécutable de patch 8 v1.1, pas
  encore en v1.3)
  `benchmark-lead --resume <run_id>` skip les étapes dont l'output existe ET est
  frais. Utile si une session Claude Code plante au milieu d'un run long.
  **Effort : M-L**. **Décision à re-évaluer après un vrai plantage en production** — si
  les runs durent < 30 min en pratique, le `--resume` apporte peu de valeur vs
  relancer from scratch.

## 2. Doctrine `benchmark-lead.md` — delta 158 lignes entre main et v1.1.x

### Contexte

`agents/benchmark-lead.md` fait **86 lignes sur main v1.2.2** (reflété par la
doctrine Sprint 1 "mainthread orchestre, benchmark-lead cadre/consolide") et
**244 lignes sur v1.1.1-fixpack** (doctrine v1.0 "benchmark-lead orchestre les
9 phases"). Soit **158 lignes de delta**.

### Analyse des 158 lignes

Ce que les 158 lignes v1.1.x contiennent et que main n'a pas :

#### Portable en v1.4 (utile)

- **Templates de prompts de cadrage plus riches** (~40 lignes) — questions
  standards à poser à l'utilisateur en cadrage (Q1-Q8 classiques). v1.3 n'en
  a qu'un stub (la section "Q5 dealbreakers" ajoutée par patch 2). **À porter
  en v1.4** pour rendre le cadrage moins improvisé.

- **Matrice "mode → livrables autorisés"** (~15 lignes) — formalise quel mode
  (Quick/Standard/Full) produit quel type de livrable. **Légère redondance**
  avec `CLAUDE.md` Sprint 1 mais documentation locale utile.

#### Dead code — à abandonner

- **Doctrine "benchmark-lead dispatch 9 phases en parallèle"** (~60 lignes) —
  impossible techniquement (limite native Claude Code : sub-agents ne
  dispatchent pas d'autres sub-agents). Le Sprint 1 v1.1.0 a corrigé ça et la
  contrainte n'a pas bougé en 2026. **Ne jamais réintroduire**.

- **Doctrine "phase X termine, phase X+1 commence automatiquement"** (~25
  lignes) — implique un orchestrateur persistant dans le sub-agent, ce qui
  n'existe pas. Le mainthread doit relancer chaque Agent call.

- **Description détaillée du handshake sub-agent ↔ scope-clarifier** (~18
  lignes) — dépendant de l'agent scope-clarifier dédié (que v1.3 Q3 a décidé
  de fusionner dans benchmark-lead). Dead code.

### Recommandation v1.4

Ne pas faire un "port complet" des 158 lignes. **Extraire ~55 lignes portables**
(templates de cadrage, matrice mode/livrables) et les intégrer ciblées dans
`agents/benchmark-lead.md`. Les 103 autres lignes restent dead code.

**Effort : S (30-45 min)** pour l'extraction + adaptation + review.

## 3. Autres artefacts des branches hardening non adressés

### 3.1 `install.py` cross-platform (bonus `341f3d8d8d`)

v1.1-hardening remplace `install.sh` + `install.bat` par un unique `install.py`
cross-platform (supporte `--no-external`, `--claude-home`). main garde les
scripts shell.

**Argument pour porter** : un seul fichier à maintenir, robustesse sur Windows
(où `install.bat` a des limites — encoding, paths avec espaces, etc.).

**Argument contre** : `install.sh` + `install.bat` fonctionnent aujourd'hui
et ont été testés (hotfix Sprint 2 a corrigé un bug Windows cp1252). Remplacer
introduit un risque de régression.

**Décision proposée v1.4** : **ne pas faire**. Conserver les scripts shell.
Si le besoin ressurgit (bug Windows récurrent), reconsidérer.

### 3.2 `prompts/ascend-bench-init.py` — générateur interactif de prompt

v1.1-hardening ajoute un petit script Python qui génère interactivement un
prompt de lancement Claude Code (calcule le `run_id`, formate le prompt, le
met dans le presse-papier). **Utilité opérationnelle moyenne** — le consultant
peut écrire ce prompt à la main en 30 secondes. **Décision v1.4 : skip**.

### 3.3 `ascend-context.md.example` + COI workflow

v1.1.1-fixpack ajoute le concept d'un fichier local `ascend-context.md` qui
liste les clients actifs / partenaires / concurrents d'Ascend. `scope-clarifier`
(fonction benchmark-lead en v1.3) consulterait ce fichier pour check COI à Q8.
Le `critic` le relirait à la fin pour flagger les recommandations en tension.

**Argument pour porter v1.4** : valeur méthodologique forte (conflit d'intérêts
explicite). Le run DGFiP (test grandeur nature v1.1) a montré qu'un consultant
sous délai peut l'oublier → filet utile.

**Argument contre** : ajoute un fichier local non versionné à maintenir.

**Décision proposée v1.4** : **porter partiellement** — ajouter
`ascend-context.md.example` au repo + section CLAUDE.md "Conflits d'intérêts"
qui explique le workflow. Pas d'agent dédié (aligné avec Q3 v1.3). **Effort : S
(30 min)**.

### 3.4 `memory-bank-public/` avec des fiches pré-peuplées

v1.1-hardening `claude/...-j4rel` branche a un `memory-bank-public/` avec 4
fiches vendor (Mistral, Dydu, LightOn, Aleph Alpha) et 2 fiches sources
(alliance.numerique.gouv.fr, cyber.gouv.fr). Ces fiches sont des artefacts
d'un run de test spécifique (DGFiP IA conversationnelle) et **ne sont pas
généralisables**.

**Décision v1.4** : **skip**. Si la memory bank est portée (patch 9, cf.
§1.3), elle se remplira organiquement au fil des missions Ascend.

### 3.5 `docs/archive/` (4 fichiers de rétrospective v1.1.1)

v1.1.1-fixpack `docs/archive/` contient 4 fichiers :
- `CHANGELOG-v1.1.1.md` (CHANGELOG de la branche avant rename)
- `FIX1-DIAGNOSTIC.md` (diagnostic détaillé de fix 1)
- `PR-BODY-v1.1.1.md` (body de PR pré-rédigé)
- `REPRISE-DEMAIN.md` (memo de reprise)

Ces fichiers sont **des artefacts de travail**, pas des docs publiques. Ils
seraient redondants avec notre `docs/sprint-*-retrospective.md`.

**Décision v1.4** : **skip**. Les conserver dans les branches archives pour
référence historique, pas dans main.

### 3.6 `outputs/20260416-1800-dgfip-ia-conv-n1/` (run test complet j4rel)

Cf. analyse v1.3 — ce run DGFiP simulé est **historiquement précieux** (11
anomalies, 12 suggestions documentées) mais **ne correspond pas au format
bench.json canonique**. Migration vers `examples/dgfip-ia-conv-n1/` serait
possible mais nécessite adaptation lourde (split factsheets/*.json →
bench.actors[], reconstitution scores.json → bench.exec_summary.ranking, etc.).

**Décision v1.4** : **migration à envisager uniquement si utile comme test
fixture pour le render de nouveaux champs v1.3** (encart disclaimer si
investigation_depth majoritaire partial, colonne investigation_depth xlsx).
Sinon, conserver en archive.
